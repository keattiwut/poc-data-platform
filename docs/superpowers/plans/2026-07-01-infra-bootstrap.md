# Infra Bootstrap Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stand up the self-hosted core infrastructure (Vault, PostgreSQL, MinIO, Airflow, ClickHouse, Superset, Airbyte) via Docker Compose (Airbyte via `abctl`), with every service's credentials sourced from HashiCorp Vault — no business logic yet.

**Architecture:** A single `docker-compose.yml` at the repo root defines Vault, PostgreSQL, MinIO, Airflow (webserver + scheduler), ClickHouse, and Superset. Airbyte runs separately via `abctl local install` (local `kind` Kubernetes cluster, per ADR-0020). A `scripts/render-env-from-vault.sh` script queries Vault's KV store and writes a git-ignored `.env` file that Docker Compose reads — this is how every service gets its credentials without any credential ever being hardcoded or committed.

**Tech Stack:** Docker Compose, HashiCorp Vault (dev mode, KV v2), PostgreSQL 16, MinIO, Apache Airflow 2.10, ClickHouse, Apache Superset, Airbyte OSS via abctl, bash + curl + jq for scripting.

## Global Constraints

- Self-hosted, on-prem infrastructure via Docker Compose (ADR-0002); Airbyte specifically via `abctl` (local Kubernetes), amending ADR-0002 per ADR-0020 — every other service stays on Docker Compose.
- Core stack: Apache Airflow (orchestrator), Airbyte (extraction), dbt (transform), ClickHouse (warehouse), Apache Superset (BI) — ADR-0004.
- Every service's credentials (DB, MinIO, ClickHouse, Superset, Airflow) are sourced from HashiCorp Vault (ADR-0006) — never hardcoded in `docker-compose.yml`, never committed in a `.env` file with real values.
- No business logic (Airflow DAGs, dbt models, Superset dashboards) is built in this plan — infra bootstrap only.
- No plaintext credentials committed to the repository or baked into Compose files.
- Vault runs in dev mode (in-memory storage, auto-unseal) for this POC bootstrap — persistent storage backend and backup wiring (ADR-0018) are explicitly deferred to a later issue (10-cicd-backups-schema-drift) and are NOT in scope here.

## File Structure

- `docker-compose.yml` — Vault, PostgreSQL, MinIO, Airflow (webserver + scheduler + init), ClickHouse, Superset services.
- `.env.example` — documents every environment variable Docker Compose expects, with placeholder (non-functional) values. Committed.
- `.env` — the real, rendered values. Git-ignored, generated by `scripts/render-env-from-vault.sh`.
- `vault/seed-secrets.sh` — writes generated credentials for every other service into Vault's KV v2 store.
- `scripts/render-env-from-vault.sh` — reads secrets back out of Vault and writes `.env`.
- `scripts/verify-vault.sh`, `scripts/verify-postgres-minio.sh`, `scripts/verify-airflow.sh`, `scripts/verify-clickhouse.sh`, `scripts/verify-superset.sh`, `scripts/verify-airbyte.sh`, `scripts/verify-dbt.sh`, `scripts/verify-full-stack.sh` — one health-check script per task, plus a final combined check.
- `scripts/install-airbyte.sh` — wraps `abctl local install`.
- `dbt/payment_gateway/dbt_project.yml`, `dbt/payment_gateway/profiles.yml` — dbt project skeleton targeting ClickHouse.
- `docs/adr/0020-airbyte-via-abctl.md` — already written (see prior turn), no action needed.

- [ ] **Step 4: Write the secret-seeding script**

Create `vault/seed-secrets.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail

VAULT_ADDR="http://localhost:8200"
VAULT_TOKEN="poc-dev-root-token"

put_secret() {
  local path="$1"
  shift
  curl -sf -H "X-Vault-Token: ${VAULT_TOKEN}" \
    -H "Content-Type: application/json" \
    -X POST \
    -d "{\"data\": {$*}}" \
    "${VAULT_ADDR}/v1/secret/data/${path}" > /dev/null
  echo "Seeded secret/${path}"
}

random_password() {
  openssl rand -hex 16
}

put_secret "postgres" \
  "\"user\": \"pipeline_admin\", \"password\": \"$(random_password)\", \"host\": \"postgres\", \"port\": \"5432\", \"db\": \"pipeline\""

echo "Vault secret seeding complete."
```

Make it executable: `chmod +x vault/seed-secrets.sh`

- [ ] **Step 5: Write the env-rendering script**

Create `scripts/render-env-from-vault.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail

VAULT_ADDR="http://localhost:8200"
VAULT_TOKEN="poc-dev-root-token"

get_field() {
  local path="$1"
  local field="$2"
  curl -sf -H "X-Vault-Token: ${VAULT_TOKEN}" \
    "${VAULT_ADDR}/v1/secret/data/${path}" \
    | jq -r ".data.data.${field}"
}

{
  echo "POSTGRES_USER=$(get_field postgres user)"
  echo "POSTGRES_PASSWORD=$(get_field postgres password)"
  echo "POSTGRES_HOST=$(get_field postgres host)"
  echo "POSTGRES_PORT=$(get_field postgres port)"
  echo "POSTGRES_DB=$(get_field postgres db)"
} > .env

echo "Rendered .env from Vault secrets."
```

Make it executable: `chmod +x scripts/render-env-from-vault.sh`
