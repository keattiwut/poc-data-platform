How the Payment Gateway Data Pipeline POC Works

An animated walkthrough of the end-to-end Transaction data flow — from Bank & Partner sources, through extraction and the data lake, into the warehouse, and out to the dashboard. Every stage is labeled with what's actually built today versus what's still planned.

1 The Concept

The Problem

The Payment Gateway currently relies on a legacy crontab-based pipeline for performance reporting. Transaction data arrives from Banks (the financial institutions that authorize and settle funds) and Partners (the merchants, aggregators, and PSPs that originate transactions) through four disconnected channels: direct database, Excel drops, CSV drops, and message queues.

There's no reliable way to reconcile these sources. If a Bank reports 10,000 transactions and a Partner reports 9,950 for the same day, nobody can say — automatically — whether that's a real discrepancy, a delayed arrival, or a duplicate-removal artifact. Nobody can confidently cite volume, Authorization Rate, Settlement Rate, or revenue without manual cross-checking. It doesn't scale.

The Solution

This POC replaces the legacy pipeline with a modernized, fully open-source platform: Airbyte extracts data from all four source types, Airflow orchestrates the daily batch run, a MinIO data lake stores bronze (raw) and silver (cleaned) Parquet zones, dbt reconciles Bank-side and Partner-side records on a shared Transaction ID and applies the Fee Schedule, ClickHouse serves as the analytical warehouse, and Superset renders the dashboards business stakeholders actually read.

The result: a single source of truth for transaction performance — with the reliability and operational maturity expected of a payment-adjacent system, even at POC scale.

IMPLEMENTED Apache Airflow — Orchestrator

Runs across every stage above (cross-cutting, not a single stop in the pipeline). Schedules the daily batch run and would trigger extraction and dbt tasks. In Issue 01 the webserver and scheduler are live, but the airflow/dags directory is an empty placeholder — no DAGs exist yet.

IMPLEMENTED HashiCorp Vault — Secrets

Feeds credentials to every service above at render time (not live runtime reads): seed-secrets.sh writes secrets, render-env-from-vault.sh generates .env, Docker Compose injects it at startup. Dev-mode in-memory storage — a Vault restart means resetting data volumes.
