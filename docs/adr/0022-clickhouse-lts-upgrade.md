# Upgrade ClickHouse to a current LTS and retire the s3-view join workaround

**Status: Accepted** (proposed in the 2026-07-09 project review — see `review_recommendation.md`; verified and implemented 2026-07-10, see Outcome below)

The stack pins `clickhouse-server:24.10`, a non-LTS release with a real bug documented (and verified against the running instance) in `int_reconciled_transactions.sql` and `fct_transactions.sql`: joins involving `s3()`-backed views silently return NULL for an unpredictable subset of columns. The workaround snapshots each staging view into a physical MergeTree table (`int_partner_transactions`, `int_bank_transactions`) before joining — two extra full copies of the data per run, plus load-bearing "don't touch this" comments that every future model has to respect.

Upgrade to a current ClickHouse LTS release, re-run the documented repro (join the `s3()`-backed staging views directly), and if the bug is gone, delete the two `int_*` snapshot models and point the reconciliation join at the staging views. The alternative — staying pinned on 24.10 because it's verified-working — was rejected: the pin trades a one-time, well-scripted verification (`verify-fct-transactions.sh` and friends already exist to prove correctness after the swap) for permanent structural complexity and an EOL release in a payment-adjacent system. Note the repro must actually be re-run, not assumed fixed; the bug's column-count sensitivity means a partial fix could pass a trivial test and still bite.

Consequence: one image-tag bump, one verification pass, and (if the repro passes) two deleted models and a shorter lineage. This also unblocks re-evaluating whether bronze→silver promotion still needs DuckDB (`scripts/promote-bronze-to-silver.py`) or can become a plain dbt model reading bronze via `s3()` — a follow-up decision, not part of this one. Relates to [ADR-0013](./0013-gold-is-clickhouse-not-lake-parquet.md) and [ADR-0014](./0014-incremental-merge-with-lookback.md).

**Outcome (2026-07-10):** `scripts/verify-clickhouse-s3-join-bug.sh` **PASSED** against the running 26.3 instance — the direct FULL OUTER JOIN of the `s3()`-backed staging views preserved all 185 matched rows and every previously-affected column (`bank_id` both sides, `decline_reason`, `bank_decline_reason`). The `int_partner_transactions` / `int_bank_transactions` snapshot models were deleted, `int_reconciled_transactions` now joins the staging views directly, and the full pipeline (via the `daily_pipeline` Airflow DAG) plus `verify-fct-transactions.sh` / `verify-dbt.sh` re-verified green. If a future ClickHouse upgrade makes the repro script fail, restore the snapshot indirection from git history.

**Progress (2026-07-09, Issue 12 — preparation done, verification pending):** `docker-compose.yml` now pins `clickhouse/clickhouse-server:26.3`, the current LTS line as of July 2026 (LTS releases are the March/August minors, supported for one year; 26.3 is the March 2026 LTS, and the previous LTS, 25.8, reaches end-of-support on 2026-08-29 — see clickhouse.com/docs/faq/operations/production and endoflife.date/clickhouse). The standalone repro exists as `scripts/verify-clickhouse-s3-join-bug.sh`. The repro has **not** been run against 26.3 — the stack was down when the bump landed — so the `int_partner_transactions` / `int_bank_transactions` workaround models are still in place. On the next stack bring-up: run the pipeline once so silver parquet exists, then run `scripts/verify-clickhouse-s3-join-bug.sh`; only on PASS delete the two `int_*` snapshot models, point `int_reconciled_transactions` at the staging views, and re-verify with `verify-fct-transactions.sh` / `verify-dbt.sh`. On FAIL, keep the workaround and record the tested version here.
