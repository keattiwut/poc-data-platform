# Adopt Airflow 3 before writing the first DAG

**Status: Accepted** (proposed in the 2026-07-09 project review — see `review_recommendation.md`; implemented in Issue 11 on Airflow 3.3.0, the current release at the time. No Airflow-3 blocker was found: the FAB auth manager ships as a bundled provider in the official image, so the existing `airflow users create` admin bootstrap kept working. Airflow 3 did require two topology additions in compose: a dedicated `dag-processor` service — the scheduler no longer parses DAGs — and a shared `AIRFLOW__API_AUTH__JWT_SECRET` plus `AIRFLOW__CORE__EXECUTION_API_SERVER_URL` so LocalExecutor tasks can reach the api-server's execution API.)

The stack pins Airflow 2.10.3 ([ADR-0004](./0004-core-oss-tool-stack.md)), but `airflow/dags/` is still empty — the orchestrator that motivates this whole project (PRD story 12: replace crontab) currently orchestrates nothing while shell scripts run the pipeline by hand. Meanwhile Airflow 3 has been GA since April 2025 and the 2.x line is winding down, so any DAG written today against 2.10 is written against a major version already scheduled for migration.

Because the DAG count is zero, this is the one moment the major-version upgrade is free: adopt Airflow 3 (same LocalExecutor/Postgres-metadata topology, updated image tag and the small config renames) before Issue 04 wires the first real DAG. The alternative — write DAGs on 2.10 now and migrate later — was rejected because it converts a config-only change into a code migration (Airflow 3 removes some 2.x DAG-authoring APIs), for no offsetting benefit. Cosmos (the planned dbt integration, ADR-0004) supports Airflow 3, so that dependency doesn't block this.

Consequence: bump the three `apache/airflow` images in `docker-compose.yml`, re-run `verify-airflow.sh`, and write all DAGs against the Airflow 3 API from day one. If a required provider turns out not to support Airflow 3 yet, staying on 2.10.x is the documented fallback — but that should be discovered by trying, not assumed.
