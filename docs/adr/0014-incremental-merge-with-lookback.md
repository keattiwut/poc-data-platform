# Incremental merge with lookback window for fct_transactions

`fct_transactions` uses dbt's `incremental_strategy: merge`, keyed on Transaction ID, with the incremental filter re-scanning a lookback window (e.g. the last 3-7 days of source data) rather than strictly "only rows since the last run." This exists specifically to handle late-arriving milestones: a transaction initiated one day can be authorized, captured, or settled several days later, and a naive "since last run" incremental filter would miss that update entirely, leaving the fact row stuck showing an old, incomplete state. Do not "simplify" this to a plain since-last-run filter — that would silently reintroduce stale fact rows for any transaction whose lifecycle spans the lookback window.

**ClickHouse can't do real in-place UPDATE/MERGE**, so this "merge" is realized via the `ReplacingMergeTree` engine (ORDER BY Transaction ID): each upsert is a new insert of the full row, and ClickHouse deduplicates to the latest version asynchronously in the background. Because the raw table can transiently hold duplicate/stale versions before that background merge runs, nothing queries `fct_transactions` directly — a dedicated dbt-built view/table (`fct_transactions_current`) selects the latest version per Transaction ID via `argMax()`, and Superset (and any downstream dbt models) point at that instead. This was chosen over having every query add `FINAL` explicitly, which has a real per-query performance cost and is an easy detail to forget when building a new dashboard.

**Implementation status (2026-07-09, Issue 12):** as built since Issue 03, `fct_transactions` is `materialized='table'` — a full rebuild on every dbt run — not the incremental merge described above (the `ReplacingMergeTree` engine and the `fct_transactions_current` dedup view do exist as described). At current POC volume a full rebuild is cheaper and simpler than incremental bookkeeping, so this is deliberate, not accidental drift. The incremental merge + lookback-window strategy above is the recorded plan for when volume warrants it; it has not yet been implemented, and the late-arriving-milestone warning still applies whenever it is.
