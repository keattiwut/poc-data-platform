# Gateway-assigned Transaction ID for cross-source reconciliation

The gateway generates a unique Transaction ID at the moment a transaction is initiated, and requires it to propagate through to both the Partner-side record and the Bank-side record. Reconciling a transaction across sources in dbt is then a simple join on that shared ID, rather than fuzzy-matching on amount/timestamp/account reference. This was chosen over fuzzy matching because it eliminates a real class of reconciliation risk (false matches, ambiguous orphans) that a performance dashboard used for company review can't afford to get wrong — but it only works because the gateway controls transaction creation and can enforce the ID's propagation. If a future Bank or Partner integration can't carry the gateway's ID through their system, that source will need a fallback fuzzy-match strategy, revisit at that point.
