Payment Gateway · Data Platform POC

One trustworthy answer to “how are we actually performing?”

Banks and Partners report the same transactions through four different channels — direct database, CSV files dropped on SFTP, and a Kafka queue. This platform extracts all of them nightly, reconciles both sides of every transaction by its gateway ID, computes fees, and serves a single dashboard the business can trust. Every module below is clickable, every claim has a verify script, and the whole stack runs from one docker compose file.

Module reference

How each layer earns its place

Grouped by what the business gets from it. Every card names its implementation and the ADR that recorded the decision — nothing here is undocumented behavior.

Sources

Mock generator & channel routing

Six Partners and four Banks, each with a stable profile (auth rate, decline mix, volume weight) and one fixed integration channel. Every simulated day produces ~1,500 correlated transaction pairs — plus deliberate anomalies (orphans, duplicate IDs, missing files) so the tests downstream have something real to catch. First run backfills 45 days; a manual reset_mock_data DAG wipes and reseeds everything.

ADR-0010 · mock/generate_transactions.py · airflow/dags/mock_data_producer.py
Extraction

dlt pipelines, one per channel

Plain Python replacing an entire Airbyte/Kubernetes deployment (ADR-0024). Each channel is its own pipeline writing Parquet to bronze: Postgres full-refresh, SFTP CSVs re-read each run, Kafka drained incrementally with offsets in dlt state. Schema contracts freeze columns and types — drift fails loudly. A PAN-shaped value in any field refuses the whole batch before it touches the lake.

ADR-0024/0019/0015 · scripts/extract-to-bronze.py · scripts/kafka_source/
Lake

Medallion zones on MinIO

Bronze holds raw per-channel Parquet (bronze/<channel>/<table>/); the promotion step deduplicates by transaction ID keeping the latest updated_at, writing one clean file per table to silver. The bucket is SSE-S3 encrypted, and each consumer authenticates with its own least-privilege user.

ADR-0017 · scripts/promote-bronze-to-silver.py · docker/minio/setup-service-users.sh
Transform

dbt: reconciliation, dimensions, 38 tests

A FULL OUTER JOIN reconciles the Partner and Bank views of every transaction (orphans survive by design); fees compute at capture from the seed schedule. Four conformed dimensions with deterministic surrogate keys feed a mart that every chart shares. dbt build runs the models and all tests in one pass, and source freshness alarms before the 6–8 AM SLA window closes.

ADR-0007/0012 · dbt/payment_gateway/models/ · models/sources.yml
Warehouse

ClickHouse on an encrypted disk

Staging views read silver straight from MinIO over TLS via the s3() table function — the warehouse is fully rebuildable from the lake at any time. All MergeTree data lands on an AES-256 encrypted disk by default policy; the query surface for dbt and Superset is HTTPS only.

ADR-0017/0022 · clickhouse/config/storage-encryption.xml · tls.xml
Serve

Superset behind the only public door

Twelve charts across the four PRD pillars — volume (gross/net), authorization and settlement rates, revenue, per-Bank/Partner comparisons and decline breakdowns, week-over-week via the date dimension. Everything renders in ~1–2 s. The one externally reachable endpoint is an nginx TLS proxy with a rate-limited login; every other UI binds to loopback.

ADR-0016 · scripts/configure-superset-dashboard.py · nginx/superset-proxy.conf
Trust

Vault-first credentials

Every secret in the platform — service passwords, encryption keys, per-service lake users — originates in Vault and reaches containers through one render step. There is no credential outside this flow, and the render fails loudly if any field is missing.

ADR-0006/0023 · vault/seed-secrets.sh · scripts/render-env-from-vault.sh
Watch

Observability with proof it fires

Prometheus scrapes ClickHouse, MinIO, and Airflow; Loki aggregates every container's logs; Grafana correlates both. Alerts route by severity to two Teams channels from two directions — Alertmanager rules and Airflow failure callbacks — each carrying a runbook link. Stopping ClickHouse really does produce the Critical card in ~2 minutes.

ADR-0008 · prometheus/rules.yml · airflow/dags/alerting.py · docs/runbooks/
Recover

Backups that restore

A cron container backs up Postgres, ClickHouse (native BACKUP), the whole lake, and Vault's storage daily at 03:00 with 14-day retention. The verify script doesn't check that backups exist — it restores all four systems and compares, including standing up a scratch Vault from the tarball and reading a secret out of it.

ADR-0018 · scripts/backup-all.sh · scripts/verify-backups.sh

Try the solution

From zero to dashboard, then break it on purpose

Prerequisites: Docker Desktop, Git Bash (or any POSIX shell), Python 3 with paramiko kafka-python psycopg2-binary duckdb requests dbt-core dbt-clickhouse. One script does the whole sequence — TLS material, Vault init/unseal, secret seeding, credential render, full stack, per-service health checks.

bring-up
$ git clone https://github.com/keattiwut/poc-data-platform.git && cd poc-data-platform
$ ./scripts/verify-full-stack.sh
# certs → vault init/unseal → seed → render .env → compose up → health checks
# first boot: allow ~5 min while the Airflow scheduler installs task deps

# after any reboot, Vault comes back sealed — recovery is always just:
$ docker compose up -d && ./vault/init-unseal.sh

The producer's first run backfills 45 days of history across all four channels; the pipeline extracts, promotes, reconciles, tests, and the dashboard fills in. Or skip the DAGs and drive it end-to-end from one script.

run
$ docker compose exec airflow-scheduler airflow dags trigger mock_data_producer
# first run: 45-day backfill (~67k transactions) into Postgres, SFTP, Kafka
$ docker compose exec airflow-scheduler airflow dags trigger daily_pipeline
# 4 dlt extracts → 2 promotions → dbt build (38 tests) → freshness check

# or the whole thing, generator to dashboard, with assertions at every step:
$ ./scripts/verify-walking-skeleton.sh

Then open the dashboard (http://localhost:8088, credentials from grep SUPERSET .env) — the volume chart's total will match the warehouse row count exactly; the verify script checks that too.

The platform's defenses are demonstrable, not decorative. Each demo injects a real failure, watches the right alarm fire on the Critical channel, and recovers.

break-it demos
$ ./scripts/verify-schema-drift.sh
# drops a CSV with a surprise column → extraction fails the schema contract
# → Critical Teams alert with runbook link → recovers after removal
$ ./scripts/verify-dq-tests.sh
# lets the generator's duplicate IDs through → the unique test catches them
$ ./scripts/verify-observability.sh
# stops ClickHouse for real → ClickHouseUnreachable fires within ~2 min
$ ./scripts/verify-security.sh
# PAN-carrying file refused · encryption verified · rate limit returns 429

# watch the alert channels while any of these run:
$ curl -s http://localhost:18080/messages | python -m json.tool

The full verification battery is nine scripts; the operator handbook indexes them all. A clean slate for demos is one DAG away — it wipes all five systems (including the Kafka offset state that naive resets forget) and reseeds a fresh backfill.

verify & reset
$ ./scripts/verify-backups.sh
# restores Postgres, ClickHouse, a lake object, and a scratch Vault — and checks them
$ docker compose exec airflow-scheduler airflow dags trigger reset_mock_data
# wipe mock DB + SFTP + Kafka + dlt state + lake + warehouse → reseed → re-run

# tear down (volumes survive); add -v only if you want to erase everything
$ docker compose down
